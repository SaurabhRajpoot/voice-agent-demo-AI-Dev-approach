## The Big Picture — Why Scaffolding Matters

### The House Analogy
Before you wire electricity or install plumbing, you pour the foundation and frame the walls. Scaffolding is that frame. It tells everyone — you, Cursor, Railway, Python — how the project is organised and what it needs.

### Three Things We Created

**1. `pyproject.toml` — The Project's Contract**
This is Python's modern way of declaring "here is what this project is and what it needs to run." It replaced the old `requirements.txt` approach. Every package the agent needs is listed here. When someone new picks up this project (or when Railway deploys it), they read this file to know what to install.

**2. `uv.lock` — The Reproducibility Guarantee**
When you run `uv sync`, uv reads `pyproject.toml`, resolves every package and *its own* dependencies, and writes the exact versions to `uv.lock`. Railway reads this file during cloud deployment — so the exact same packages that work on your MacBook get installed in the cloud. No "works on my machine" surprises. This file MUST be committed to git.

**3. Folder Structure — The Map**
A specific layout so every file has a logical home. The `customers/` folder is the heart of the configurable-by-design principle — every client demo is just a folder inside it. The top-level code never hardcodes a client name; it reads `ACTIVE_CUSTOMER` from the environment and loads the right folder.

## Every File and Folder Explained

### `pyproject.toml`
The single most important file in a modern Python project. Contains:
- Project name, version, description
- Required Python version (`>=3.12`)
- All production dependencies (packages needed to run)
- All dev dependencies (packages only needed during development)

```toml
[project]
name = "voice-agent-demo"
version = "0.1.0"
description = "AI voice pre-sales agent"
requires-python = ">=3.12"
dependencies = [
    "livekit-agents~=1.0",
    "livekit-plugins-deepgram",
    "livekit-plugins-elevenlabs",
    "livekit-plugins-openai",
    "livekit-plugins-silero",
    "qdrant-client",
    "openai",
    "pyyaml",
    "python-dotenv",
]

[dependency-groups]
dev = [
    "pytest",
    "pytest-asyncio",
]
```

**Why `~=1.0` for livekit-agents?**
The `~=` operator means "compatible with version 1.0" — it allows patch updates (1.0.1, 1.0.2) but blocks major version jumps (2.0). You get bug fixes automatically without risking breaking changes.

**Why separate dev dependencies?**
`pytest` is only needed on your MacBook during development. Railway doesn't need it to run the agent. Separating them keeps the production Docker image smaller and faster.

---

### `uv.lock`
Auto-generated by `uv sync`. You never edit this manually. It's a detailed record of every package and its exact version — including the dependencies of your dependencies. At 386KB, it covers all 90 packages that `livekit-agents` and its plugins need.

**The lockfile concept:** When you add a package, uv records the exact version in `uv.lock`. When Railway deploys your code, it reads that lockfile and installs *exactly the same versions* — not "latest compatible", but byte-for-byte identical. This is what makes "it works on my MacBook AND in production" reliable.

> **Rule:** Always commit `uv.lock` to git. Railway reads it for reproducible builds.

## Key Concepts Learned

### uv vs pip — Why uv

| Problem with pip | How uv solves it |
|---|---|
| Slow installs | uv is 10–100x faster |
| No lockfile by default | uv generates `uv.lock` automatically |
| "Works on my machine" | `uv.lock` guarantees identical packages everywhere |
| Inconsistent dev vs cloud | Railway reads `uv.lock` — same versions always |

**Key rule:** Never type `pip install` in this project. Always `uv add` or `uv sync`.

---

### Virtual Environments — What `.venv` Is
A virtual environment is an isolated copy of Python with its own package library. Without it, every Python project on your Mac shares the same packages, and version conflicts become a nightmare. With `.venv`, this project has its own clean Python installation. `uv` creates and manages it automatically.

---

Cursor has three main modes:

| **Agent Mode** | `Cmd+I` | Creating multiple files, running terminal commands, multi-step tasks |
| **Inline Edit** | `Cmd+K` | Editing a specific section of an existing file |

**Why Agent Mode for Chat 1?** This chat creates ~6 files across multiple directories in one go. Agent Mode can read your project, run terminal commands, create files, and verify results — all from one instruction. You'll use Inline Edit heavily from Chat 4 onwards when refining existing files.

---

### The Configurable-by-Design Principle
The most important architectural principle in this build. Before hardcoding anything, ask:

> "Would this value need to change if I switched to the MBIE demo?"

If yes → it goes in `customers/{name}/config.yml`. Never in the code.

This is what makes switching customers as simple as changing one environment variable (`ACTIVE_CUSTOMER=mbie`) with zero code changes.

## What Cursor Did — Step by Step

Here is exactly what Cursor did behind the scenes when it ran that prompt. This is your "understanding what happened" section.

---

### STEP 1 — `uv init --python 3.12`

**What Cursor ran:**
```bash
uv init --python 3.12
```

**What this command does:**
- Creates `pyproject.toml` with project metadata
- Creates `.python-version` file containing `3.12` — pins this project to Python 3.12
- Creates a stub `main.py` (hello world)
- Creates an empty `README.md`
- Initialises a git repository (`.git/` folder) — makes the project version-controllable

**Why Python 3.12 specifically?**
livekit-agents 1.x requires Python 3.12+. The `--python 3.12` flag tells uv to use this version for the project's virtual environment. The `.python-version` file means any tool that reads it (uv, pyenv, Railway) uses the same version.

---

### STEP 2 — Create Folder Structure

**What Cursor ran:**
```bash
mkdir -p customers/rockgas/docs
mkdir -p tests
```

**What `-p` means:** Creates nested directories in one command, including all parents that don't exist yet. Without `-p`, you'd have to create `customers/`, then `customers/rockgas/`, then `customers/rockgas/docs/` as three separate commands.

**Why this structure?**
- `customers/rockgas/` — everything specific to the RockGas demo lives here
- `customers/rockgas/docs/` — KB content files for ingestion (Day 2)
- `tests/` — pytest discovers tests here automatically

---

### STEP 3 — Update `pyproject.toml` with Dependencies

**What Cursor did:**
Edited `pyproject.toml` to add all packages under `[project]` → `dependencies` and dev packages under `[dependency-groups]` → `dev`.

**Each package and why it's needed:**

| Package | Why |
|---|---|
| `livekit-agents~=1.0` | The core agent framework — orchestrates STT, LLM, TTS pipeline |
| `livekit-plugins-deepgram` | Deepgram STT integration for livekit-agents |
| `livekit-plugins-elevenlabs` | ElevenLabs TTS integration for livekit-agents |
| `livekit-plugins-openai` | OpenAI LLM + TTS integration for livekit-agents |
| `livekit-plugins-silero` | Silero VAD (Voice Activity Detection) — detects when caller starts/stops speaking |
| `qdrant-client` | Python client for Qdrant vector database |
| `openai` | OpenAI SDK — used directly for embeddings (not via livekit plugin) |
| `pyyaml` | Reads `config.yml` files |
| `python-dotenv` | Loads `.env` file into environment variables |
| `pytest` (dev) | Test runner |
| `pytest-asyncio` (dev) | Lets pytest run async Python test functions |

---

### STEP 4 — Create `.env.example`

**What Cursor did:**
Created a new file `.env.example` with all key names and blank values.

**Why blank values?**
This file is committed to git. If it had real values, those secrets would be public. The blank template tells any developer "you need these variables — fill them in yourself from your own accounts."

---

### STEP 5 — Create `.gitignore`

**What Cursor did:**
Created `.gitignore` with entries for Python standard ignores plus `.env` and `.DS_Store`.

**Key entries and why:**
```
.venv/          ← Virtual env is huge (386KB lockfile, ~200MB actual packages)
                   It's reproducible from uv.lock — no need to commit it
__pycache__/    ← Python's compiled cache — auto-generated, platform-specific
*.pyc           ← Compiled Python files — same reason
.env            ← CRITICAL: real API keys — never commit
.DS_Store       ← macOS folder metadata — meaningless to other developers
dist/           ← Build output — not source code
```

---

### STEP 6 — Create `AGENTS.md`

**What Cursor did:**
Created a markdown file summarising the project for AI assistants.

**Why this matters for Cursor specifically:**
Every time you open a new Cursor Agent session, it reads your project files for context. `AGENTS.md` gives it an instant briefing — the stack, the services, the conventions — without you having to re-explain in every prompt. It's like a standing memo for your AI coding partner.

---

### STEP 7 — `uv sync`

**What Cursor ran:**
```bash
uv sync
```

**What this command does, in order:**
1. Reads `pyproject.toml` to find all declared dependencies
2. Resolves the full dependency tree (every package, plus their dependencies, plus their dependencies' dependencies)
3. Writes exact pinned versions to `uv.lock`
4. Creates `.venv/` if it doesn't exist
5. Installs all packages into `.venv/`
6. Result: 90 packages installed, including `livekit-agents==1.4.3`

**Why this took 1–3 minutes:**
First time uv runs, it downloads packages from PyPI (Python's package repository). Subsequent `uv sync` runs are much faster because packages are cached.

## How To Do It Manually (Without Cursor)

If you ever need to set up a Python project like this yourself from scratch — no Cursor, no prompts — here are the exact manual steps.

### Prerequisites
```bash
# Install uv (if not already installed)
curl -LsSf https://astral.sh/uv/install.sh | sh
source $HOME/.local/bin/env

# Verify
uv --version
```

### Step 1: Create and enter project folder
```bash
mkdir voice-agent-demo
cd voice-agent-demo
```

### Step 2: Initialise the project
```bash
uv init --python 3.12
```
This creates: `pyproject.toml`, `.python-version`, `main.py`, `README.md`, `.git/`

### Step 3: Create folder structure
```bash
mkdir -p customers/rockgas/docs
mkdir -p tests
```

### Step 4: Add production dependencies
```bash
uv add "livekit-agents~=1.0"
uv add livekit-plugins-deepgram
uv add livekit-plugins-elevenlabs
uv add livekit-plugins-openai
uv add livekit-plugins-silero
uv add qdrant-client
uv add openai
uv add pyyaml
uv add python-dotenv
```

Each `uv add` updates `pyproject.toml` and `uv.lock` automatically.

### Step 5: Add dev dependencies
```bash
uv add --dev pytest
uv add --dev pytest-asyncio
```

### Step 6: Create `.env.example`
```bash
cat > .env.example << 'EOF'
ACTIVE_CUSTOMER=
LIVEKIT_URL=
LIVEKIT_API_KEY=
LIVEKIT_API_SECRET=
DEEPGRAM_API_KEY=
ELEVEN_API_KEY=
OPENAI_API_KEY=
QDRANT_URL=
QDRANT_API_KEY=
GMAIL_APP_PASSWORD=
EOF
```

### Step 7: Create `.gitignore`
```bash
cat > .gitignore << 'EOF'
.venv/
__pycache__/
*.pyc
*.pyo
dist/
build/
*.egg-info/
.env
.DS_Store
EOF
```

### Step 8: Create your actual `.env` with real keys
```bash
cp .env.example .env
# Then open .env in any text editor and fill in real values
```

### Step 9: Sync all packages
```bash
uv sync
```

### Step 10: Verify
```bash
ls -la                    # see all files
uv run python --version   # should show 3.12.x
ls customers/rockgas/     # should show docs/
ls uv.lock                # should exist
```

### Step 11: Commit to git (safe files only)
```bash
git add pyproject.toml uv.lock .env.example .gitignore AGENTS.md README.md
git add customers/
git add tests/
git commit -m "Initial project scaffold"
# Note: .env is automatically excluded by .gitignore
```
